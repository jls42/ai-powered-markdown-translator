- **1.9** Correzione del silent-failure + strumentazione qualità completa + nota di traduzione multi-posizione (2026-05-07) :
  - **Nota di traduzione multi-posizione + formato marker "embed card"** :
    - Nuove opzioni CLI (aggiuntive, default invariati → **non breaking**) :
      - `--note_position {top,bottom,both}` (default : `bottom`) : posiziona la nota in alto, in basso, o in entrambi i punti del file tradotto.
      - `--note_format {legacy,marker}` (default : `legacy`) :
        - `legacy` riproduce rigorosamente il comportamento v1.8 (paragrafo in grassetto `**…**`) **byte-for-byte**.
        - `marker` emette una link reference definition Markdown invisibile (`[ai-translation-note-<placement>]: <> "v=1 source=… target=… model=… date=…"`) seguita da un **blockquote di 3 paragrafi** strutturato per un rendering tipo "GitHub repo embed card" : titolo del progetto in inline code (`**\`ai-powered-markdown-translator\`\*\*`), descrizione tradotta dal LLM, e link CTA (`[Voir le projet sur GitHub ↗](URL)`) con freccia visibile. Utilizzabile in build tramite un plugin remark (cf. blog jls42.org → plugin `remark-translation-banner`).
    - **Invarianti mai inviate al LLM** : titolo del repo e URL GitHub vengono assemblati lato Python dopo la traduzione della frase descrittiva. Il LLM non vede mai né lo slug `ai-powered-markdown-translator` né `https://github.com/jls42/...`, garantendo che nessun renderer/maiuscole-scheme venga alterato.
    - **Inserimento consapevole del frontmatter** : in modalità `top` o `both`, la nota viene inserita **dopo il blocco di chiusura `---`** del frontmatter YAML (sicurezza Astro Content Collections / gray-matter). Il helper `_split_frontmatter` rileva `---\n…\n---\n` all'inizio del file e ne preserva l'integrità ; **solleva `RuntimeError`** su frontmatter aperto senza fence di chiusura (il file risale in `failed_files` invece di essere scritto con una nota mal posizionata).
    - **Sanitizer modello whitelist** : `_sanitize_model` sostituisce ogni carattere fuori `[A-Za-z0-9._:/-]` con `_`, fallback `unknown` se vuoto. Si allinea al validatore lato plugin remark Astro e neutralizza i caratteri che romperebbero il formato del marker (spazio, virgolette, parentesi, virgola, ecc.).
    - **Refactor interno** : `_append_translation_note` (1 funzione monolitica) → 7 helper puri (`_translation_note_invariants`, `_build_translation_note_phrase`, `_assemble_translation_note_paragraphs`, `_build_translation_note_source`, `_sanitize_model`, `_quote_lines`, `_split_frontmatter`, `_build_translation_note_block`, `_compose_with_notes`). Builder/composer separati (il builder restituisce un blocco puro senza separatore, il composer applica i `\n\n` secondo la posizione) ; produzione e helper source condividono lo stesso assemblatore di 3 paragrafi.
    - **`_quote_lines` blank-preserving** : prefissa ogni riga con `> `, trasformando le righe vuote in `>` solo. Permette a mdast di vedere 3 paragrafi distinti nel blockquote (titolo / descrizione / link) invece di un solo paragrafo con line-break.
    - **`_build_translation_note_block` adattivo** : in base al numero di paragrafi che il LLM ha preservato (3 = formato card completo, 2 = frase + link, 1 = fallback). Il fallback a 1 paragrafo **non avvolge più in `**...**`** quando viene rilevato un link Markdown `](` (rendering fragile di `<strong>` intorno a un link).
    - **Compatibilità retroattiva** : `getattr(args, "note_position", "bottom")` e `getattr(args, "note_format", "legacy")` lato `_compose_with_notes` — i Namespace senza questi attributi (test esistenti, chiamate programmatiche esterne) continuano a funzionare senza modifica.
  - **Fix silent-failure sulle traduzioni lunghe** :
    - Validazione lingua post-traduzione su tutti i provider (OpenAI, Mistral, Claude, Gemini) : livello deterministico (estratto sorgente ritrovato verbatim) + livello probabilistico (`langdetect`)
    - Whitelist `finish_reason` / `stop_reason` : sollevare `RuntimeError` su ogni stato fuori dalla whitelist (truncation, content_filter, ecc.)
    - `max_tokens` Claude : `4096` → `32768` (evita truncation latente su segmenti 16k, margine cross-script FR→JA/ZH/KO/AR/HI)
    - Segmentazione heading-aware : priorità H2/H3 nella seconda metà del segmento (ogni segmento inizia con una sezione semantica completa)
    - Propagazione degli errori fino a exit code non-zero : `translate_markdown_file` restituisce uno stato tipizzato `success` / `failure` / `skipped`, `main()` `sys.exit(1)` se almeno un file è fallito (single-file e batch)
    - Empty-content guard su tutti i provider, sanity ratio source/output (≥ 500 char, < 5% = rifiuto), validazione placeholder code (`#CODEBLOCK`/`#INLINECODE`), normalizzazione post-LLM (separatori/link attaccati a un heading), `BadRequestError` retry senza `reasoning_effort`
