- **1.9.1** Corrección de i18n de la etiqueta CTA en la nota de traducción marker (2026-05-10) :

  - **Bug corregido**: la etiqueta `[Voir le projet sur GitHub ↗]` del enlace CTA en el banner marker de la parte superior de los archivos traducidos seguía estando **en francés** para todos los idiomas de destino, en lugar de seguir `target_lang`. El LLM nunca la ve (se ensambló del lado de Python para preservar la URL y el slug del repo), así que la fase de traducción no podía recuperarla. Regresión silenciosa desde la incorporación del formato `marker` en v1.9.
  - **Corrección**: nueva constante `_VIEW_PROJECT_LABELS` que mapea los 15 idiomas a su etiqueta localizada. `_translation_note_invariants(target_lang)` y `_assemble_translation_note_paragraphs(phrase, target_lang)` propagan ahora el idioma de destino. Fallback `fr` si el idioma es desconocido (seguridad, sin KeyError).
  - **Tests**: `test_source_emits_three_paragraphs_repo_title_description_link` ajustado (target_lang `ja` → etiqueta japonesa esperada). 2 nuevos tests: `test_source_link_label_localized_per_target_lang` (parametrizado sobre 7 idiomas que cubren scripts latino, ideográfico y abjad) y `test_source_link_label_falls_back_to_french_for_unknown_target`. Total: 40 tests en `test_translation_note_position.py` (en lugar de 38).
  - **Compatibilidad retrospectiva**: firma con valor por defecto `target_lang="fr"` — los llamadores programáticos externos sin `args.target_lang` siguen funcionando sin cambios.

- **1.9** Corrección de silent-failure + conjunto completo de herramientas de calidad + nota de traducción multiposición (2026-05-07) :
  - **Nota de traducción multiposición + formato marker "embed card"** :
    - Nuevas opciones CLI (aditivas, valores predeterminados sin cambios → **no breaking**) :
      - `--note_position {top,bottom,both}` (predeterminado: `bottom`) : coloca la nota arriba, abajo, o en ambos lugares del archivo traducido.
      - `--note_format {legacy,marker}` (predeterminado: `legacy`) :
        - `legacy` reproduce estrictamente el comportamiento v1.8 (párrafo en negrita `**…**`) **byte por byte**.
        - `marker` emite una definición de referencia de enlace Markdown invisible (`[ai-translation-note-<placement>]: <> "v=1 source=… target=… model=… date=…"`) seguida de un **blockquote de 3 párrafos** estructurado para un renderizado tipo "GitHub repo embed card" : título del proyecto en código inline (`**\`ai-powered-markdown-translator\`\*\*`), descripción traducida por el LLM, y enlace CTA (`[Voir le projet sur GitHub ↗](URL)`) con flecha visible. Aprovechable en build mediante un plugin remark (cf. blog jls42.org → plugin `remark-translation-banner`).
    - **Invariantes nunca enviadas al LLM** : el título del repo y la URL de GitHub se ensamblan del lado de Python después de traducir la frase descriptiva. El LLM nunca ve el slug `ai-powered-markdown-translator` ni `https://github.com/jls42/...`, garantizando que ningún renderer/mayúscula-minúscula/esquema se altere.
    - **Inserción consciente de frontmatter** : en modo `top` o `both`, la nota se inserta **después del bloque de cierre `---`** del frontmatter YAML (seguridad Astro Content Collections / gray-matter). El helper `_split_frontmatter` detecta `---\n…\n---\n` al inicio del archivo y preserva su integridad; **lanza `RuntimeError`** sobre un frontmatter abierto sin fence de cierre (el archivo vuelve a subir en `failed_files` en lugar de escribirse con una nota mal colocada).
    - **Sanitizador de modelo whitelist** : `_sanitize_model` reemplaza todo carácter fuera de `[A-Za-z0-9._:/-]` por `_`, fallback `unknown` si está vacío. Se alinea con el validador del lado del plugin remark de Astro y neutraliza los caracteres que romperían el formato del marker (espacio, comillas, paréntesis, coma, etc.).
    - **Refactor interno** : `_append_translation_note` (1 función monolítica) → 7 helpers puros (`_translation_note_invariants`, `_build_translation_note_phrase`, `_assemble_translation_note_paragraphs`, `_build_translation_note_source`, `_sanitize_model`, `_quote_lines`, `_split_frontmatter`, `_build_translation_note_block`, `_compose_with_notes`). Builder/composer separados (el builder devuelve un bloque puro sin separador, el composer aplica los `\n\n` según la posición); producción y helper source comparten el mismo ensamblador de 3 párrafos.
    - **`_quote_lines` blank-preserving** : prefija cada línea con `> `, transformando las líneas vacías en `>` solamente. Permite a mdast ver 3 párrafos distintos en el blockquote (título / descripción / enlace) en lugar de un solo párrafo con line-breaks.
    - **`_build_translation_note_block` adaptativo** : según el número de párrafos que el LLM haya preservado (3 = formato card completo, 2 = frase + enlace, 1 = fallback). El fallback de 1 párrafo **ya no se envuelve en `**...**`** cuando se detecta un enlace Markdown `](` (renderizado frágil de `<strong>` alrededor de un enlace).
    - **Compatibilidad ascendente** : `getattr(args, "note_position", "bottom")` y `getattr(args, "note_format", "legacy")` del lado de `_compose_with_notes` — los Namespace sin estos atributos (tests existentes, llamadas programáticas externas) siguen funcionando sin cambios.
  - **Corrección de silent-failure en traducciones largas** :
    - Validación de idioma post-traducción en todos los providers (OpenAI, Mistral, Claude, Gemini): capa determinista (extracto fuente reencontrado verbatim) + capa probabilística (`langdetect`)
    - Whitelist `finish_reason` / `stop_reason` : lanzar `RuntimeError` ante cualquier estado fuera de la whitelist (truncation, content_filter, etc.)
    - `max_tokens` Claude : `4096` → `32768` (evita truncation latente en segmentos 16k, margen cross-script FR→JA/ZH/KO/AR/HI)
    - Segmentación heading-aware: prioridad H2/H3 en la segunda mitad del segmento (cada segmento comienza con una sección semántica completa)
    - Propagación de los errores hasta un exit code no cero: `translate_markdown_file` devuelve un estado tipado `success` / `failure` / `skipped`, `main()` `sys.exit(1)` si al menos un archivo ha fallado (single-file y batch)
    - Empty-content guard en todos los providers, sanity ratio source/output (≥ 500 chars, < 5% = rechazo), validación de placeholders code (`#CODEBLOCK`/`#INLINECODE`), normalización post-LLM (separadores/enlaces pegados a un heading), `BadRequestError` retry sin `reasoning_effort`
    - Añadida dependencia `langdetect==1.0.9`
  - **Herramientas de calidad pre-commit** ("tipo EurekAI completo", 14 hooks) :
